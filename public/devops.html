Executive Summary

Juan's World is a full-stack web application built on Next.js, Upstash Redis, and Vercel that serves as a personal publishing platform with multi-user access controls, AI agent integration, and multilingual content delivery.

The system enables AI agents to autonomously publish markdown content that renders to live webpages, deployed serverlessly on Vercel with Redis as primary storage and filesystem fallback for data durability.

Executive Summary

What This System Is

Juan's World represents a modern approach to web architecture that collapses traditional frontend-backend divisions through Next.js —a framework allowing developers to write both presentation and logic layers in unified JavaScript/TypeScript. For someone who has never encountered Next.js, imagine combining two separate jobs: the frontend developer who creates what users see (HTML, CSS, JavaScript) and the backend developer who manages servers, databases, and business logic. Next.js enables a single developer or team to handle both aspects seamlessly.

Key Architectural Decisions:

  • Static HTML frontend for instant page loads and CDN distribution
  • API Routes for dynamic functionality and AI agent integration
  • Upstash Redis as primary in-memory database for performance
  • Filesystem fallback for data durability during Redis outages
  • Vercel hosting for serverless deployment and global distribution

What It Does

Juan's World serves four distinct audiences with tailored experiences. Public visitors access static HTML pages (index, about, diary) with instant loading times. Authenticated users receive personalized dashboards with content filtered by assignment—seeing only reports and STL files explicitly shared with them. Administrators manage the platform through an admin panel, controlling user accounts, generating API keys, and uploading 3D assets. AI agents autonomously publish content through API key authentication, enabling automated content workflows.

Why This Architecture Works

The design reflects deliberate trade-offs prioritizing simplicity and operational resilience over enterprise complexity. Static pages eliminate server processing for most requests, reducing costs and improving performance. The Redis-plus-filesystem model provides exceptional speed with data safety nets. Dual authentication methods (HMAC sessions for humans, API keys for agents) acknowledge different interaction patterns. Role-based access control with four distinct roles hits the sweet spot: simple enough to implement correctly, expressive enough to enforce real security boundaries.

Core Concepts Explained

Static HTML Frontend vs. API Routes

What Static HTML Means

In Juan's World, static HTML frontend refers to pre-built HTML files served directly from Vercel's CDN without server-side processing. Unlike dynamic sites that assemble pages per-request, these files are generated once during deployment and served instantly worldwide.

Performance Benefits:

  • • Speed: No database queries or template rendering
  • • Cost: Static file serving is essentially free at scale
  • • Reliability: No server failures or memory leaks
  • • Global Access: CDN caches at hundreds of edge locations

What API Routes Provide

API routes are special Next.js files that define HTTP endpoints. They live in the app/api/ directory and handle raw HTTP requests rather than rendering pages.

// Example API Route (app/api/content/route.ts) export async function GET(request: Request) { const { searchParams } = new URL(request.url); const category = searchParams.get('category'); // ... query Redis, return JSON return Response.json({ items: [...] }); }

Think of it this way:

Static HTML pages are the "display layer" —what everyone sees. API routes are the "action layer" —handling logins, saving content, and serving dynamic data. Every interactive feature on a static page ultimately depends on an API route behind the scenes.

Redis as Primary Database with Filesystem Fallback

Why Redis Instead of Traditional Databases

Redis is an in-memory data structure store that serves as Juan's World's primary database. Unlike PostgreSQL or MySQL, Redis keeps all data in RAM, providing extraordinary performance—hundreds of thousands of operations per second.

✓ Advantages:

  • • Sub-millisecond read latency
  • • Simple key-value/hash data model
  • • Managed service (no server maintenance)
  • • Perfect for read-heavy content sites

⚠ Considerations:

  • • Data volatility (RAM-based)
  • • Limited query capabilities (no SQL joins)
  • • Managed service costs

The Filesystem Fallback Safety Net

The filesystem fallback ensures data durability when Redis is unavailable. Every write follows a cascade:

Write Flow: 1. Update in-memory cache (immediate availability) 2. Try Redis (primary storage) 3. If Redis fails → Try local filesystem (backup) 4. If filesystem fails → Keep in memory (ephemeral)

On local development: Files written to ./data/ and ./content/ (persistent)
On Vercel: Files written to /tmp/ (ephemeral)

Session-Based Authentication Using HMAC Signatures

What Sessions Solve

Sessions allow users to prove identity once (via password) and then access protected resources for days without re-authentication. Without sessions, every request would require resending credentials.

HMAC: Stateless Verification

Rather than storing random tokens in a database, Juan's World uses HMAC (Hash-based Message Authentication Code) to generate cryptographically signed tokens.

// Session Token Format: username:signature // Signature = SHA256(username:SESSION_SECRET).slice(0,32) // Example: alice:a3f7b2d8e9c1... // Verification requires no database lookup: function checkAuth(token) { const [username, sig] = token.split(':'); const expected = SHA256(username + SECRET).digest('hex').slice(0,32); return sig === expected ? username : null; }

Key Security Properties:

  • ✓ Stateless: No database queries during verification
  • ✓ Fast: Microsecond validation time
  • ✓ Secure: Signature depends on server-side secret
  • ✓ Scoped: Token contains username directly

API Key Authentication for AI Agents

Why Agents Need Different Authentication

AI agents are software programs, not human users. They cannot use session cookies because they don't run in browsers, don't maintain cookie jars across requests, and don't interact with HTML login forms.

Human Users:

  • • Session cookies
  • • Browser-managed
  • • 7-day expiration
  • • Broad permissions

AI Agents:

  • • API keys
  • • Programmatic
  • • Until revoked
  • • Narrow permissions

API Key Validation Flow

API keys are validated through lookup in the Redis-stored api-keys array. The validation process:

// Agent Request Headers: { "Content-Type": "application/json", "x-api-key": "ak_live_51H8m..." } // Validation in lib/api-keys.ts: function validateApiKey(headerKey) { const validKeys = JSON.parse(await redis.get('api-keys') || '[]'); return validKeys.some(keyObj => keyObj.key === headerKey); }

Architecture Walkthrough

Reading the diagram from top to bottom follows a typical request through the system:

  1. Users Layer: Four actor types initiate interactions through different authentication methods
  2. Frontend Layer: Static HTML pages serve as entry points, with language variants branching horizontally
  3. API Layer: Serverless functions handle dynamic requests through two patterns: JavaScript fetch() and direct form POSTs
  4. Backend Libraries: Reusable modules centralize complex logic like authentication, content processing, and audit logging
  5. Storage Layer: Dual-write pattern ensures data safety with Redis primary storage and filesystem fallback
  6. External Services: Third-party integrations handle email delivery and analytics monitoring

Key Insight:

The architecture's strength lies in its clean separation of concerns—each layer has specific responsibilities and well-defined interfaces with adjacent layers. This modularity enables independent development, testing, and scaling of components.

Data Flow Walkthroughs

1. Public Visitor Reads a Diary Entry

Step 1

Browser Requests diary.html

Visitor types juan.world/diary.html. Vercel's CDN responds instantly with pre-built static file—no server processing, no authentication required.

Step 2

JavaScript Calls /api/content

Embedded JavaScript executes fetch() to /api/content?category=update&lang=en. No credentials needed for public content.

Step 3

API Route Queries Redis

Handler calls scanCategory('update', 'en'), which sends HGETALL content:update:en to Upstash Redis. Sub-millisecond response time.

Step 4

Content Processing Pipeline

For each entry: gray-matter parses YAML frontmatter, marked renders markdown to HTML, isPublic filter removes private content.

Step 5

Browser Renders Content

JSON response travels back to browser. JavaScript injects rendered HTML into page. Complete process typically under 1 second.

2. A Logged-In User Authenticates

Step 1

User Submits Credentials

User enters username/password on login.html. JavaScript intercepts form submission and sends fetch() to /api/login.

Step 2

Password Verification

lib/users.ts verifies password using PBKDF2 with stored salt. Intentionally slow (hundreds of ms) to deter brute force attacks.

Step 3

HMAC Session Creation

lib/auth.ts creates token: username:SHA256(username:SESSION_SECRET).slice(0,32). No database write needed—cryptographic signature alone proves authenticity.

Step 4

Secure Cookie Set

Server responds with Set-Cookie: session=token; HttpOnly; Secure; SameSite=Strict; Max-Age=604800. Browser stores cookie automatically.

Step 5

Automatic Authentication

Browser includes session cookie with every subsequent request. Server verifies HMAC signature statelessly (microseconds) and applies role-based content filtering.

3. An AI Agent Publishes New Content

Step 1

Agent Sends PUT Request

AI agent constructs HTTP PUT to /api/content/upload with x-api-key header and JSON payload containing slug, title, content, category, lang, isPublic.

Step 2

API Key Validation

lib/api-keys.ts validates key by looking up in Redis-stored api-keys array. Returns 401 Unauthorized if invalid, 403 Forbidden if revoked.

Step 3

Content Sanitization

Agent-provided frontmatter is stripped using gray-matter to prevent metadata injection. System rebuilds clean YAML frontmatter from validated fields.

Step 4

Dual-Layer Storage

lib/content-db.ts attempts Redis HSET first. If Redis fails, writes to filesystem (local: ./content/, Vercel: /tmp/). Returns storage location in response.

Step 5

Immediate Live Content

Content is immediately available via public API. Next visitor refreshing diary.html will see the new entry. Agent receives success response with slug and storage location.

Overall Security Model

Juan's World implements defense in depth with explicit role separation , designed to minimize attack surface while maintaining operational simplicity.

Defense Layers:

  • • Static HTML eliminates server-side execution risks
  • • Separate auth mechanisms prevent credential confusion
  • • Role-based filtering at data retrieval point
  • • Immutable logging provides audit trails

Security Principles:

  • • Never trust the client—server-side validation only
  • • Fail closed—absence of credentials = rejection
  • • Principle of least privilege for AI agents
  • • Cryptographic rather than database-dependent auth

Why Redis Was Chosen

  • Performance: Sub-millisecond read latency ensures responsive user experience
  • Simplicity: Key-value/hash structures map directly to JavaScript objects
  • Managed Service: Upstash eliminates operational burden
  • Cost Model: Serverless pricing aligns with Vercel's pay-per-use model
  • Natural Caching: In-memory operation provides built-in hot data caching

Filesystem Fallback Protection

  • Transient Outage Protection: Handles brief Redis unavailability
  • Development Flexibility: Enables local development without Redis
  • Disaster Recovery: Files can be used to reconstruct Redis state
  • Data Portability: Markdown files remain human-readable and tool-agnostic
  • Layered Reliability: Multiple fallback paths reduce data loss probability

Content API Endpoints

GET /api/content

Public & Authenticated

Query Parameters:

  • category (required): "update", "brief", or "report"
  • lang (optional, default "en"): "en", "ja", or "es"
  • slug (optional): specific post identifier

Authentication:

  • • None required for public content
  • • Session cookie for private content access

Response Shape:

{ "items": [ { "slug": "day-11-breakthrough", "title": "Day 11 — Breakthrough", "html": "

...

", "date": "2024-01-15T14:32:00Z", "isPublic": true } ] }

Description: Retrieves content entries for specified category and language. Public visitors receive only isPublic: true entries. Authenticated users additionally receive their assigned private content. Admins receive all entries.

PUT /api/content/upload

AI Agent Only

Request Body:

{ "slug": "day-12-progress", "title": "Day 12 — Making Progress", "content": "# Updates\n\nSignificant milestones...", "category": "update", "lang": "en", "isPublic": true }

Authentication:

  • • x-api-key header required
  • • Session cookies rejected

Response Shape:

{ "success": true, "slug": "day-12-progress", "storage": "redis" }

Validation:

  • • Slug validated for URL safety
  • • Embedded frontmatter stripped
  • • Clean YAML frontmatter rebuilt

Description: Creates new content entry with validated metadata. Strips any agent-provided frontmatter to prevent injection attacks. Stores in Redis hash with filesystem fallback. Returns storage location for monitoring.

Auth API Endpoints

POST /api/login

Public

Request Body:

{ "username": "alice", "password": "wonderland123" }

Security Features:

  • • HTTPS encryption in transit
  • • PBKDF2 password hashing
  • • Slow verification to deter brute force

Response Headers:

Set-Cookie: session=alice:a3f7b2d8e9c1...; HttpOnly; Secure; SameSite=Strict; Max-Age=604800

Cookie Attributes:

  • • HttpOnly: JavaScript cannot read
  • • Secure: HTTPS only
  • • SameSite: CSRF protection

Description: Verifies password against PBKDF2 hash, creates HMAC-signed session token, sets secure cookie with 7-day expiration, records login event in audit log, updates user's last login timestamp.

POST /api/logout

Authenticated

Request:

  • • Requires valid session cookie
  • • No request body needed
  • • Cookie identifies session to destroy

Response Headers:

Set-Cookie: session=; Max-Age=0; Expires=Thu, 01 Jan 1970 00:00:00 GMT

Stateless Nature:

  • • No database deletion needed
  • • Simply removing cookie invalidates session
  • • HMAC verification will fail without cookie

Description: Invalidates current session by clearing cookie. Due to stateless HMAC verification, simply removing the cookie prevents future authentication. Records logout event in audit log if implemented.

Admin API Endpoints

GET /api/users

Admin Only

Authentication:

  • • Valid session cookie required
  • • User must have role: "admin"
  • • 403 Forbidden otherwise

Security Filtering:

  • • Password hashes excluded from response
  • • Only safe fields serialized

Response Shape:

{ "users": [ { "username": "alice", "role": "user", "createdAt": "...", "lastLogin": "..." } ] }

Description: Returns all user accounts for admin review. Sensitive fields like passwordHash are stripped before serialization. Requires admin role; non-admin requests receive 403 Forbidden.

POST /api/keys

Admin Only

Request Body:

{ "label": "Morning Brief Bot" }

Response:

  • • Returns key exactly once
  • • Cannot be retrieved later
  • • If lost, must revoke and regenerate

Response Shape:

{ "success": true, "key": "ak_live_51H8m...", "label": "Morning Brief Bot" }

Key Characteristics:

  • • Cryptographically random
  • • Bearer token format
  • • Stored with creation date

Description: Generates new cryptographically random API key, stores in api-keys array with label and creation date, returns key exactly once. Key cannot be retrieved later; if lost, must be revoked and regenerated.

Environment Variables Setup Guide

Step-by-step instructions for configuring environment variables, with explanations of each variable's purpose, where to obtain values, and what breaks if they're missing.

Required Variables

UPSTASH_REDIS_REST_URL

Required

What it is:

HTTPS endpoint URL for Upstash Redis REST API (format: https://us1-allowed-mollusk-12345.upstash.io)

Where to get it:

  1. 1. Create Upstash account at upstash.com
  2. 2. Click "Create Database"
  3. 3. Select region near your Vercel deployment
  4. 4. Go to "REST API" section
  5. 5. Copy UPSTASH_REDIS_REST_URL

What happens if missing:

  • • All database operations fail immediately
  • • Cannot read users, content, API keys
  • • Login attempts return 500 errors
  • • Filesystem fallback activates but limited

Critical: Application becomes non-functional without this variable.

UPSTASH_REDIS_REST_TOKEN

Required

What it is:

Bearer token for authenticating REST API requests to Upstash Redis (format: AYPdASQgNTY3MD..., long base64 string)

Where to get it:

  1. 1. Same Upstash dashboard "REST API" section
  2. 2. Copy UPSTASH_REDIS_REST_TOKEN
  3. 3. Distinct from Redis CLI password
  4. 4. Never share publicly—full database access

What happens if missing:

  • • Redis operations fail with authentication errors
  • • Cannot establish database connectivity
  • • Same impact as missing URL variable

SESSION_SECRET

Required

What it is:

Cryptographically random string used to sign and verify HMAC session tokens (64-character hex recommended)

How to generate:

# Using Node.js: node -e "console.log(require('crypto').randomBytes(32).toString('hex'))" # Or using OpenSSL: openssl rand -hex 32

What happens if missing:

  • • Session creation fails with error
  • • Users cannot log in
  • • Existing sessions become invalid
  • • Entire authenticated experience broken

Auto-Detected Variables

VERCEL

Auto-Set

What it is:

Boolean flag automatically set to "1" by Vercel's build and runtime environment

How it's set:

  • • Automatically provided by Vercel
  • • Do not set manually
  • • Verify in build logs if debugging

What it controls:

  • • Filesystem path switching
  • • VERCEL=1: uses /tmp/ (ephemeral)
  • • VERCEL unset: uses ./data/ (persistent)
  • • Enables identical code in both environments

Important: This explains why files "disappear" on Vercel—they're in /tmp/ which is ephemeral.

Glossary

Definitions of all technical terms used in this guide, explained in plain English for beginners.

Infrastructure & Platform

Next.js

A React framework that combines frontend and backend development in a single codebase, enabling both static site generation and server-side rendering.

App Router

Next.js 13+'s routing system using the app/ directory, supporting nested layouts, server components, and API routes.

API Route

Special file that defines an HTTP endpoint rather than rendering HTML pages, handling raw requests and responses.

Vercel

Cloud platform optimized for hosting Next.js applications with built-in CDN, serverless functions, and automatic deployment.

CDN

Content Delivery Network - geographically distributed servers that cache and serve static files from locations near users.

Security & Authentication

HMAC

Hash-based Message Authentication Code - cryptographic technique combining secret key with hash function to verify data authenticity.

PBKDF2

Password-Based Key Derivation Function 2 - algorithm that intentionally slows down password hashing to deter brute force attacks.

Session Cookie

Small piece of data stored by browser and automatically sent with requests, containing authentication token in this case.

API Key

Long random string used by software programs to authenticate API requests, enabling machine-to-machine communication.

RBAC

Role-Based Access Control - security model where permissions are assigned to roles rather than individual users.

Content & Data

Markdown

Lightweight markup language using plain text formatting syntax (e.g., # for headings, ** for bold) that separates content structure from presentation.

Frontmatter

YAML metadata placed at beginning of markdown files between --- lines, storing structured data like title, date, and visibility.

Slug

URL-safe identifier derived from title, typically lowercase with hyphens replacing spaces, used as unique key for content entries.

Static HTML

Pre-built HTML files served directly without server-side processing, providing fastest possible delivery and CDN compatibility.

SSR

Server-Side Rendering - generating HTML on server for each request, enabling dynamic content but requiring more compute resources.

Integration & Tools

Resend

Email delivery service with developer-friendly API, used in Juan's World for contact form submissions.

Upstash

Managed Redis hosting service with serverless pricing and REST API access, eliminating operational burden of running Redis servers.

Google Analytics 4

Web analytics service that tracks visitor behavior, page views, and engagement metrics across the application.

gray-matter

JavaScript library that parses YAML frontmatter from markdown files, separating metadata from content body.

marked

JavaScript library that converts markdown syntax to HTML, rendering headings, paragraphs, links, and other markdown elements.

Storage & Caching

Redis

In-memory data structure store used as database, cache, and message broker, providing exceptional performance for read-heavy workloads.

In-Memory Cache

Temporary data storage in RAM, faster than disk or network access but lost on process restart, used for sub-millisecond responses.

Filesystem Fallback

Resilience mechanism that writes data to local disk when primary Redis store is unavailable, providing backup storage path.
